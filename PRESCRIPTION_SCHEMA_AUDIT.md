## 2. Should doctor_name remain required after Phase 2?

### ❌ NO.

After Phase 2, there are **two distinct prescription types** in the same table:

### Type A: COFEPRIS Prescription (POS-created)
```
patient_name        = "Juan Pérez"          (manual entry, required)
doctor_name         = "Dr. García"          (manual entry, often left empty)
doctor_license      = "12345678"            (manual entry, often left empty)
prescription_number = "MANUAL-123456"       (auto or manual)
prescription_date   = "2026-06-05"          (required)
sale_id             = <sale UUID>           (linked to POS sale)
doctor_id           = NULL                  (not a doctor prescription)
customer_id         = may be NULL           (walk-in vs registered)
```

### Type B: Doctor Prescription (Doctor Portal-created)
```
patient_name        = "Juan Pérez"          (auto from customer.full_name, required)
doctor_name         = NULL or ""            (NOT needed — doctor_id FK is canonical)
doctor_license      = NULL or ""            (NOT needed — stored in doctor_profiles)
prescription_number = "RX-20260616-00001"   (auto-generated by trigger)
prescription_date   = "2026-06-05"          (required)
sale_id             = NULL                  (unfilled until POS links it)
doctor_id           = <doctor profile UUID> (FK to profiles, required)
customer_id         = <customer UUID>       (FK to customers, required)
medication          = "Paracetamol 500mg"   (clinical field)
dosage              = "1 tableta"           (clinical field)
```

### Why doctor_name must be nullable:

1. **Doctor prescriptions use `doctor_id` (FK)** — the doctor's identity is relational, not free-text
2. **Doctor's license is in `doctor_profiles.license_number`** — duplicating it in `doctor_license_number` is denormalized and error-prone
3. **The COFEPRIS modal already passes `null`** — `PrescriptionModal.jsx:112` does `doctor_name: formData.doctorName.trim() || null`. Even COFEPRIS prescriptions fail if the user skips the optional doctor fields.
4. **Same applies to `doctor_license_number`** — `PrescriptionModal.jsx:113` does `doctor_license_number: formData.doctorLicense.trim() || null`

### What should remain NOT NULL:

| Column | Status | Reason |
|--------|--------|--------|
| `patient_name` | **KEEP NOT NULL** | Always populated by both COFEPRIS modal and PatientWorkspace |
| `prescription_number` | **KEEP NOT NULL** | RX trigger auto-generates before constraint check; COFEPRIS modal always provides it |
| `prescription_date` | **KEEP NOT NULL** | Always populated by both code paths |
| `doctor_name` | **MAKE NULLABLE** | Redundant when `doctor_id` is present; optional for COFEPRIS |
| `doctor_license_number` | **MAKE NULLABLE** | Redundant when `doctor_id` is present; optional for COFEPRIS |

## 3. Final Schema of the prescriptions Table (After Both Migrations + Fix)

```sql
CREATE TABLE prescriptions (
  -- Primary key
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),

  -- Relationships
  sale_id uuid REFERENCES sales(id) ON DELETE CASCADE,      -- NULL until POS links sale
  org_id uuid REFERENCES organizations(id),                   -- Required for RLS
  doctor_id uuid REFERENCES profiles(id),                     -- FK to doctor profile (Type B)
  customer_id uuid REFERENCES customers(id),                  -- FK to patient (Type B)

  -- Patient info (always required)
  patient_name text NOT NULL,
  patient_curp text,

  -- Doctor info (legacy COFEPRIS fields — now nullable)
  doctor_name text,                                           -- WAS: NOT NULL
  doctor_license_number text,                                 -- WAS: NOT NULL
  doctor_office_address text,
  doctor_phone text,

  -- Prescription identification
  prescription_number text NOT NULL,                          -- Auto by trigger or manual
  prescription_date date NOT NULL,

  -- Clinical fields (Type B doctor prescriptions)
  medication text,
  dosage text,
  frequency text,
  duration text,
  notes text,

  -- Lifecycle
  status text DEFAULT 'active'
    CHECK (status IN ('active','fulfilled','expired','cancelled')),
  expires_at date,
  fulfilled_at timestamptz,

  -- COFEPRIS voiding
  is_voided boolean DEFAULT false,
  voided_at timestamptz,
  voided_by uuid REFERENCES profiles(id),

  -- Audit
  created_at timestamptz DEFAULT now(),
  created_by uuid REFERENCES profiles(id),
  prescribed_by uuid REFERENCES profiles(id),                 -- Who created the prescription

  -- Indexes
  UNIQUE(sale_id) WHERE sale_id IS NOT NULL,                  -- Partial unique: 1 prescription per sale
);
```

### Indexes (after all migrations):

```sql
-- From original schema
idx_prescriptions_date ON prescriptions(prescription_date)

-- From MIGRATION_P2_extend_prescriptions.sql
idx_prescriptions_number ON prescriptions(prescription_number)
idx_prescriptions_customer ON prescriptions(customer_id) WHERE customer_id IS NOT NULL
idx_prescriptions_status ON prescriptions(status)
idx_prescriptions_doctor ON prescriptions(doctor_id) WHERE doctor_id IS NOT NULL
idx_prescriptions_sale_unique ON prescriptions(sale_id) WHERE sale_id IS NOT NULL
```

## 6. Root Cause

The original `prescriptions` table (from `PHASE1_SCHEMA.sql`) was designed exclusively for COFEPRIS compliance records where a POS operator manually typed in doctor name and license. When Phase 2 added doctor-created prescriptions with a proper `doctor_id` foreign key, the legacy `NOT NULL` constraints were never revisited. The P2 migration added new columns but did not relax the old constraints.

Additionally, the POS `PrescriptionModal.jsx` already treats `doctor_name` and `doctor_license_number` as optional UI fields (passing `null` when empty), which means the existing COFEPRIS flow had a **latent bug** that would surface whenever staff skipped those optional fields.
